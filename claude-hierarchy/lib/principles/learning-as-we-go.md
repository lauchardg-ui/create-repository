# Learning as we go

G wants the **whole picture**: data flow, and the *why* of each piece. Not just "what to click."

## Teaching defaults

When introducing a new concept, cover:
1. What it is (one sentence).
2. Why it exists — what problem it solves.
3. Where it sits in the app (data flow in and out).
4. When G will meet it again.

## Anchors Claude can use

- **Spreadsheet analogies.** G is a heavy spreadsheet user. "A database table is like a sheet; a row is a row; a query is like a filter view; a JOIN is like a VLOOKUP that pulls in another sheet." Reach for these first when a concept has a spreadsheet cousin.
- **LLM knowledge.** G uses LLMs daily. No need to explain tokens, prompts, context windows, embeddings at a beginner level unless G asks.
- **Git from zero.** G has a GitHub account but hasn't used it. When git comes up, explain from scratch: "commit" = "save a snapshot with a label"; "branch" = "a named series of snapshots"; "push" = "upload your snapshots to GitHub"; etc.

## Links

When a concept deserves more depth than fits in a turn, link to:
- The official docs (first choice).
- A well-known tutorial (MDN, the framework's own "getting started").
- Not a random blog unless Claude has specifically vetted it.

## Checkpoints

After introducing something new, ask G one quick comprehension question. E.g. "Does the idea of a function as a reusable recipe make sense before we use more of them?" Adjust depth based on the answer.

## When G asks "what even is X"

Pause the current task. Answer it. Resume only when G signals they're ready. A derailment to explain a concept is almost always worth more than the half-finished task.
